### 로그 #1

| 항목 | 내용 |
|---|---|
| 일시 | 2026-06-04 |
| 단계 | 구현·검토 (M3) |
| 사용 도구 | Claude |

**프롬프트 (요약):**
구현된 `src/main.py` 코드를 PM 입장에서 간단히 검토하려 한다. 클래스 다이어그램과 코드 구조가 일치하는지 확인하는 방법과, PM이 비개발 역할로도 할 수 있는 코드 검토 관점을 알려 달라고 요청하였다.

**AI 응답 요약:**
> 클래스 이름, 메서드 이름, 주요 속성이 설계 문서(클래스 다이어그램)와 일치하는지 비교하는 방법을 제시하였다. PM이 코드 세부 구현보다는 '요구사항 충족 여부', '클래스 이름·구조의 설계 문서 일치 여부', '실행 가능 여부'를 중심으로 검토하면 된다고 설명하였다.

**비판적 검토:**
> PM 관점의 검토 포인트 제안은 유용하였다. 다만 "코드 실행 후 모든 시나리오를 직접 테스트하라"는 제안은 QA/보안 역할(안광우)과 중복되는 영역이어서 PM이 직접 수행할 필요가 없다고 판단하였다. 역할 분담 원칙에 따라 PM은 구조 확인에 집중하기로 하였다.

**팀/개인 수정 내용:**
> AI 제안 중 '요구사항 충족 여부'와 '설계 문서 대비 클래스 구조 일치 여부' 두 가지 관점만 채택하였다. 실제 코드 검토는 `src/main.py`를 열어 7개 클래스명과 주요 메서드를 클래스 다이어그램과 직접 비교하는 방식으로 수행하였다. `Quiz.answers` 속성 누락은 이미 임태훈이 수정한 상태임을 확인하였다.

**최종 활용 결과:**
> 회의록 #9 작성 시 코드 검토 내용 정리에 반영하였다. PM 역할로 구현 산출물을 검토하는 기준 수립에 활용하였다.

### 로그 #2

| 항목 | 내용 |
|---|---|
| 일시 | 2026-06-15 |
| 단계 | 마무리 (M3) |
| 사용 도구 | Claude |

**프롬프트 (요약):**
소프트웨어 공학 개론 팀 프로젝트 최종보고서에 포함되어야 할 13개 항목 목록을 제시하고, 누락 항목이나 구성 순서 문제가 있는지 검토해 달라고 요청하였다.

**AI 응답 요약:**
> 13개 항목 구성이 전반적으로 적절하다고 평가하였다. 추가로 '리스크 관리 결과', '테스트 시나리오 결과표' 항목을 보완하면 더 완성도 있는 보고서가 된다고 제안하였다. 항목 순서는 현재 구성이 논리적으로 자연스럽다고 하였다.

**비판적 검토:**
> 추가 제안 항목('리스크 관리', '테스트 결과표')은 과목 제출 가이드(PHASE3-6)에 명시되지 않은 항목이었다. 가이드 외의 항목을 추가하면 보고서 구성이 가이드 기준에서 벗어날 수 있다고 판단하였다. AI가 일반적인 SW공학 보고서 기준으로 제안한 것이므로, 과목 맥락에서는 제외하는 것이 적절하였다.

**팀/개인 수정 내용:**
> 항목 추가 제안은 채택하지 않고, 가이드 기준 13개 항목 그대로 유지하기로 결정하였다. AI 검토를 통해 현재 목차에 누락이 없다는 것을 확인하는 용도로만 활용하였다.

**최종 활용 결과:**
> `M3_최종보고서_단어즈.md` 목차 구성의 이중 검증에 활용하였다. 실제 항목 구성은 가이드 기준을 그대로 따랐다.

### 로그 #3

| 항목 | 내용 |
|---|---|
| 일시 | 2026-06-16 |
| 단계 | 마무리 (M3) |
| 사용 도구 | ChatGPT |

**프롬프트 (요약):**
팀 프로젝트 회고를 '잘된 점·아쉬운 점·배운 점·다음에 다시 한다면' 구성으로 작성하려 한다. 회고 문서에서 흔히 쓰이는 표현 방식과 주의해야 할 점을 알려 달라고 요청하였다.

**AI 응답 요약:**
> "팀워크가 좋았다", "소통이 원활했다"처럼 추상적인 표현 대신 구체적인 사건·수치를 포함하라고 조언하였다. '잘된 점'에는 팀이 잘 수행한 구체적인 행동을, '아쉬운 점'에는 다음에 개선할 수 있는 사항을 솔직하게 기술하라고 안내하였다.

**비판적 검토:**
> 방향성 자체는 제출 가이드의 조언("잘된 점만 나열하는 것보다 아쉬운 점과 배운 점을 솔직하게 쓰는 것이 좋은 평가를 받는다")과 일치하였다. 다만 AI가 제시한 예시 문장들은 어느 팀 프로젝트에도 붙일 수 있는 일반적인 표현이어서 그대로 사용하기에는 부적절하였다.

**팀/개인 수정 내용:**
> 예시 문장은 사용하지 않고, 우리 팀의 실제 경험(설계 패턴 문서 지연, Mermaid 렌더링 오류 발견 경험, 인스펙션 Cross-check 효과)을 기반으로 회고를 직접 작성하였다. 팀원별 소감도 각자에게 직접 받아 최종보고서에 반영하였다.

**최종 활용 결과:**
> `docs/M3/회고 및 개선 사항.md` 및 최종보고서 13번 항목의 작성 방향 참고에 활용하였다. 실제 내용은 팀이 직접 작성하였다.

## 성찰 일지

### AI 로그 #2 제출 시점 (14주차, M3 최종 완료)

**이번 기간 동안 AI를 어떻게 활용했나요?**
> M3 단계에서는 구현 코드 검토 관점 파악, 최종보고서 목차 이중 검증, 회고 작성 방향 참고 등 총 3회 활용하였다. M1·M2 단계의 설계·기획 작업에 비해 M3는 산출물 취합과 통합이 주를 이뤄 AI 활용 필요성이 낮았다.

**AI 활용에서 가장 도움이 된 순간은?**
> 최종보고서 목차를 검토받을 때 13개 항목에 누락이 없다는 것을 빠르게 확인할 수 있었던 점이 도움이 되었다. PM으로서 전체 문서 구성의 완성도를 점검하는 용도로 활용한 것이 가장 실질적이었다.

**AI 활용에서 가장 어렵거나 주의가 필요했던 순간은?**
> 회고 작성 시 AI 예시 표현이 제출 가이드에서 '나쁜 예'로 명시된 수준과 유사하였다. AI가 평가 기준 문서를 모르는 상태에서 일반적인 답변을 내놓는다는 것을 다시 한번 확인하였다. AI 답변을 비판 없이 수용하면 오히려 낮은 평가를 받을 수도 있다는 점에서 주의가 필요하였다.

**AI 없이 직접 해야 했던 작업은?**
> 인스펙션 결과 반영 여부 최종 판단, 팀원 소감 취합, 계획 대비 실적 주차 비교, GitHub 최종 업로드 상태 확인은 전적으로 직접 수행하였다. 팀의 실제 진행 경험과 의사결정 이력은 AI가 알 수 없기 때문이다.

**한 학기 AI 활용 전체를 돌아보며:**
> M1~M3를 거치면서 AI를 가장 효과적으로 활용한 방식은 '문서 구조나 형식의 참고'와 '누락 항목 확인'이었다. 내용 자체를 AI에게 맡기려 할수록 팀 맥락에 맞지 않는 결과물이 나왔다. AI를 '초안 생성자'가 아닌 '검토 보조자'로 활용하는 방식이 PM 역할에 가장 잘 맞았다고 느꼈다.
