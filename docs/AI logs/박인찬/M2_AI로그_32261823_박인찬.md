### 로그 #1

| 항목 | 내용 |
|---|---|
| 일시 | 2026-05-07 |
| 단계 | 설계 (M2) |
| 사용 도구 | Claude |

**프롬프트 (요약):**
소프트웨어 개발 팀 프로젝트에서 GitHub, KakaoTalk, Google Docs를 조합하여 협업 도구를 운영하는 방식을 정리하고, 실제 운영 계획 문서를 작성할 때 포함해야 할 항목을 알려 달라고 요청하였다.

**AI 응답 요약:**
> 협업 도구 문서에 포함할 항목으로 '도구 목록 및 용도', '운영 규칙(업로드 주기, 담당자)', '문제 발생 시 대응 방안'을 제안하였다. GitHub은 코드·문서 관리, KakaoTalk은 실시간 소통, Google Docs는 공동 편집 초안 작성 용도로 구분하는 예시를 제시하였다.

**비판적 검토:**
> 도구 분류 방식은 우리 팀 실제 운영 방식과 유사하여 구조 참고가 가능하였다. 다만 AI가 제시한 '운영 규칙' 항목은 "매일 업데이트"처럼 지나치게 이상적인 기준을 제시하여 현실적이지 않았다. 또한 Google Docs를 최종 문서로 제출하는 방식은 우리 과목 가이드(Markdown → GitHub 제출)와 맞지 않아 제외하였다.

**팀/개인 수정 내용:**
> 도구별 용도 구분은 참고하되, 운영 방식 설명은 실제 팀 회의에서 결정된 방식(Google Docs에서 초안 작성 후 Markdown으로 변환하여 GitHub 업로드)으로 직접 작성하였다. 업로드 주기도 "작업 완료 시마다 즉시"로 실제 팀 규칙을 반영하였다.

**최종 활용 결과:**
> `docs/M2/다음 단계 계획` 문서의 협업 도구 항목 구성 참고에 활용하였다. 실제 내용은 팀 논의 결과로 직접 작성하였다.

**프롬프트 (요약):**
GitHub에서 Mermaid 코드블록으로 작성한 유스케이스 다이어그램과 클래스 다이어그램이 정상적으로 렌더링되는지 확인하는 방법과, 렌더링 오류가 발생할 경우 주요 원인을 알려 달라고 요청하였다.

**AI 응답 요약:**
> GitHub에서 코드블록 언어를 ` ```mermaid ` 로 정확히 지정해야 렌더링된다는 것을 설명하였다. 주요 오류 원인으로 언어 지정 오타, 문법 오류(화살표 방향 혼동, 특수문자 처리 문제), 한글 노드 이름 사용 시 일부 버전에서 깨짐 현상을 제시하였다.

**비판적 검토:**
> 주요 오류 원인 설명은 실제 경험과 일치하여 유용하였다. 다만 "한글 노드 이름이 항상 문제가 된다"는 설명은 과도한 일반화였다. 실제로 GitHub 최신 버전에서는 한글 노드 이름도 렌더링이 잘 되는 것을 팀 내 테스트로 확인하였다.

**팀/개인 수정 내용:**
> 한글 노드 이름 관련 부분은 채택하지 않았다. 코드블록 언어 지정 방법과 화살표 방향 주의 사항만 팀에 공유하였다. 임태훈이 작성한 UML 파일의 클래스 다이어그램 Mermaid 코드를 GitHub에서 직접 열어 렌더링 상태를 확인하는 방식으로 검증하였다.

**최종 활용 결과:**
> 팀원(임태훈)의 UML 파일 검토 시 렌더링 오류 원인 파악에 활용하였다. `docs/M2/UML 다이어그램.md` 최종 파일의 렌더링 정상 여부 확인에 간접 기여하였다.

**프롬프트 (요약):**
소프트웨어 설계 단계가 완료된 후 구현 단계로 넘어가기 전에 PM이 점검해야 할 항목 체크리스트를 작성해 달라고 요청하였다.

**AI 응답 요약:**
> 체크리스트 항목으로 '요구사항과 설계 산출물 연계 확인', '팀원별 구현 담당 파트 명확화', '개발 환경 통일', '버전 관리 규칙 재확인', '테스트 계획 수립' 등을 제시하였다.

**비판적 검토:**
> 대부분의 항목이 실무 개발 프로젝트 기준으로 작성된 것이어서 과목 범위에 비해 과도한 내용이 포함되었다. '테스트 계획 수립' 항목은 우리 프로젝트가 별도 테스트 문서를 요구하지 않으므로 불필요하였다. PM이 확인해야 하는 핵심 항목(요구사항-설계 연계, 구현 담당자 확정, 개발 환경)만 선별하였다.

**팀/개인 수정 내용:**
> 선별한 핵심 항목을 바탕으로 팀 내 확인 사항 목록을 직접 정리하였다. Python 단일 파일 구현, 표준 라이브러리만 사용, `src/main.py` 파일명 통일 등 우리 팀 구체 사항은 AI 제안과 무관하게 팀이 직접 결정하였다.

**최종 활용 결과:**
> M2→M3 전환 시점의 팀 내 준비 확인에 활용하였다. 회의록 #8 및 `docs/M2/다음 단계 계획` 문서 내 우선순위·리스크 항목 작성에 간접 참고하였다.

## 성찰 일지

### AI 로그 #1 제출 시점 (12주차, M2 완료)

**이번 기간 동안 AI를 어떻게 활용했나요?**
> M2 단계에서는 협업 도구 문서 구성 참고, Mermaid 오류 대응, M2→M3 전환 체크리스트 작성 참고 등 총 3회 활용하였다. M1에 비해 활용 빈도가 낮았는데, 설계 내용 자체(UML, SOLID 등)는 설계자·분석가가 주도하는 영역이어서 PM 역할에서 AI 활용 필요성이 상대적으로 적었다.

**AI 활용에서 가장 도움이 된 순간은?**
> Mermaid 렌더링 오류 원인 파악 시 도움이 되었다. 팀원이 작성한 UML 파일이 GitHub에서 렌더링되지 않았을 때 빠르게 원인(화살표 문법 오류)을 찾을 수 있었다.

**AI 활용에서 가장 어렵거나 주의가 필요했던 순간은?**
> AI가 실무 수준의 체크리스트나 운영 규칙을 제안할 때, 과목 범위와 팀 규모에 맞게 걸러내는 판단이 필요하였다. AI의 제안을 기준으로 삼으면 오히려 과도한 작업이 생길 수 있다는 것을 경험하였다.

**AI 없이 직접 해야 했던 작업은?**
> 설계 검토 회의 진행, 팀원 간 의견 조율, 문서 통합 일정 관리는 직접 수행하였다. 팀의 상황과 진행 속도는 AI가 알 수 없기 때문이다.

**M3 단계에서는 AI를 어떻게 활용할 계획인가요?**
> 최종보고서 통합 시 항목 누락 여부 확인 용도로 한 번 활용할 계획이다. 내용 작성은 M1·M2 산출물을 직접 취합하여 정리하는 방식을 유지할 것이다.
