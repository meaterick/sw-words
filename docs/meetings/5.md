## 논의 내용

**안건 1. 협업 도구 운영 방식 확정**

논의 내용:
M2 단계에서 문서 작업이 늘어나니 협업 방식을 좀 더 명확히 정하자는 의견이 나왔다. 박인찬이 Claude에게 협업 도구 운영 문서에 포함할 항목을 사전에 확인해봤다. AI는 "매일 업데이트"처럼 현실적이지 않은 기준을 제안해서 실제 팀 방식으로 직접 정리했다. Google Docs는 초안 작성에만 쓰고 최종은 Markdown으로 GitHub에 올리는 것이 과목 가이드 방식이라 그대로 하기로 했다.

결정 사항:
- GitHub: 문서·코드 관리, 작업 완료 시마다 커밋
- KakaoTalk: 빠른 소통, 일정 조율
- Google Docs: 초안 작업 및 공동 편집 → 완료 후 Markdown으로 변환 업로드

---

**안건 2. 유스케이스 다이어그램 방향 논의**

논의 내용:
안병규가 가져온 유스케이스 초안을 같이 확인했다. 사용자 액터가 하나(일반 사용자)이고, 주요 유스케이스가 퀴즈 생성·풀이·오답 저장·기록 조회라는 점에서 의견이 일치했다. 유스케이스 간 include/extend 관계를 어떻게 설정할지 논의가 길어졌다. 오답 자동 저장(UC6)이 정답 확인(UC5)을 extend하는 관계로 보는 것이 자연스럽다는 결론을 냈다.

결정 사항:
- UC2(퀴즈 생성) → UC1(과목 선택) include 관계 확정
- UC3(퀴즈 풀이) → UC4(정답 제출) include 관계 확정
- UC6(오답 저장) → UC5(정답 확인) extend 관계 확정
- 다이어그램 작성 도구: Mermaid (GitHub에서 직접 렌더링)

---

**안건 3. 클래스 다이어그램 초안 작성 범위**

논의 내용:
임태훈이 Mermaid 문법은 어느 정도 파악했다고 했다. 클래스는 요구사항의 주요 객체(User, Quiz, Question, Answer, WrongAnswerNote, QuizHistory, Subject)를 기반으로 설계하기로 했다. 각 클래스의 속성과 메서드는 임태훈이 초안을 잡고, 개발자(박성주)가 구현 가능성 측면에서 검토하는 방식으로 진행.

결정 사항:
- 클래스 목록 초안: User, Subject, Quiz, Question, Answer, WrongAnswerNote, QuizHistory
- 임태훈이 다이어그램 초안을 5월 12일까지 작성 후 팀 공유
