**프롬프트 (요약):**
퀴즈 기반 학습 앱 프로젝트의 기능 요구사항 초안을 검토하고, 누락되었을 수 있는 항목을 제안해 달라고 요청하였다.

**AI 응답 요약:**
> 기존 FR 5개 항목은 적절하다고 평가하였다. 추가 제안으로 '학습 목표 설정 기능', '난이도 조절 기능', '소셜 공유 기능'을 제시하였다. 비기능 요구사항으로는 성능·보안 외에 접근성(Accessibility) 항목도 추가하면 좋겠다고 하였다.

**비판적 검토:**
> 추가 제안 항목들은 5인 팀, 약 4주 구현 일정인 우리 프로젝트 규모에 비해 과도한 범위라고 판단하였다. 접근성 NFR은 CLI 기반 앱에서 적용 범위가 불분명하여 제외하였다. AI가 일반적인 앱 기능 목록을 나열한 것으로, 프로젝트 맥락에 맞게 걸러낼 필요가 있었다.

**팀/개인 수정 내용:**
> FR은 기존 5개(퀴즈 생성·답안 제출·해설 확인·오답 저장·학습 기록)로 유지하고, NFR은 성능·보안 2개만 채택하였다. FR-04 우선순위를 '중'으로 상향 조정하는 결정은 팀 회의에서 인터뷰 결과를 바탕으로 팀이 직접 판단하였다.

**최종 활용 결과:**
> `docs/M1/requirements.md` 요구사항 정의서 검토 시 누락 항목 확인 용도로 활용하였다. 기존 항목의 완성도를 재확인하는 데 그쳤으며, AI 제안 내용은 채택하지 않았다.

**프롬프트 (요약):**
소프트웨어 개발 팀 프로젝트의 WBS를 5인 팀·10주 일정 기준으로 기획·설계·구현·마무리 단계로 나눠 예시를 작성해 달라고 요청하였다.

**AI 응답 요약:**
> 단계별 작업 항목, 담당 역할, 주차를 표 형식으로 제시하였다. 기획 단계에 '요구사항 분석', '프로토타입 제작', 구현 단계에 '단위 테스트', '통합 테스트' 등이 포함되어 있었다.

**비판적 검토:**
> AI가 제시한 WBS는 교과서 수준의 일반적인 템플릿이어서 우리 팀의 강의 주차 기준 일정과 역할 구조(PM·분석가·설계자·개발자·QA/보안)가 반영되어 있지 않았다. '프로토타입 제작', '통합 테스트' 항목은 현 프로젝트 범위와 맞지 않아 제외하였다. 주차 번호와 담당자는 실제 강의 계획서를 보고 직접 결정해야 하였다.

**팀/개인 수정 내용:**
> AI 예시의 표 구조만 참고하였다. 작업 항목·담당자·주차는 팀 회의(회의록 #3)에서 직접 조율하여 확정하였다. 제출 기준 주차도 강의 계획서를 직접 확인하여 14주차로 수정하였다.

**최종 활용 결과:**
> `docs/M1/wbs.md` WBS 문서의 표 형식 구조 참고에 활용하였다. 내용 전체는 팀이 직접 결정하였다.

GitHub 팀 저장소 README.md에 포함할 항목(팀명·역할·주요 기능·마일스톤·문서 링크)을 포함한 구조 예시를 Markdown 형식으로 작성해 달라고 요청하였다.

**AI 응답 요약:**
> 프로젝트명, 팀 구성 표, 주요 기능 목록, 마일스톤 링크, 문서 링크 섹션으로 구성된 README 예시를 Markdown 형식으로 제시하였다.

**비판적 검토:**
> 구조 자체는 적절하였으나, 프로젝트 개요 문장이 짧고 일반적이었다. GitHub 계정·마일스톤 URL은 실제 저장소 정보로 교체해야 하여 AI 예시 주소를 그대로 사용할 수 없었다. 팀원 GitHub 계정 정보도 별도로 수집하여 직접 입력이 필요하였다.

**팀/개인 수정 내용:**
> 프로젝트 개요는 요구사항 정의서의 프로젝트 소개 문구를 바탕으로 직접 작성하였다. 마일스톤 URL과 GitHub 계정은 실제 정보로 교체하였다. 섹션 구조만 AI 제안을 참고하였다.

**최종 활용 결과:**
> `README.md` 작성 시 섹션 구조 참고에 활용하였다. 내용은 팀 정보와 프로젝트 개요를 직접 작성하여 반영하였다.

> 단계별 AI 활용 목적과 방법, 크로스 체크 방식(도구 간 교차 검토 + 팀원 상호 검토)을 포함한 활용 계획 예시를 제시하였다.

**비판적 검토:**
> 원칙 문구가 "AI 결과를 그대로 사용하지 않는다"처럼 당연한 내용 위주였고, 실제로 어떻게 비판적으로 검토할 것인지에 대한 방법이 빠져 있었다. 크로스 체크 방식도 팀원 간 상호 검토와 도구 간 교차 검토를 구분 없이 서술하여 실제 운영 방식이 불분명하였다.

**팀/개인 수정 내용:**
> '비판적 검토'의 의미를 구체화하여 "논리적·기술적 결함을 확인하고 AI가 누락한 맥락을 팀이 보완한다"는 내용을 추가하였다. 크로스 체크를 GPT·Gemini·Claude 교차 활용과 팀원 상호 검토 두 가지 축으로 명확히 구분하여 작성하였다.

**최종 활용 결과:**
> `docs/M1/AI utilization plan.md`에 반영하였다. AI 초안 구조를 참고하되 문구는 팀 논의 후 직접 수정하였다.

## 성찰 일지

### AI 로그 #1 제출 시점 (8주차, M1 완료)

**이번 기간 동안 AI를 어떻게 활용했나요?**
> 주로 문서 초안의 구조·형식을 참고하거나 누락 항목을 확인하는 용도로 활용하였다. 요구사항 검토, WBS 구조 참고, README 구성 참고, AI 활용 계획 작성 등 PM 역할에서 처음 작성하는 형식의 문서마다 AI에게 예시를 요청하였다. 총 4회 활용하였다.

**AI 활용에서 가장 도움이 된 순간은?**
> 요구사항 정의서 초안을 검토할 때 "혹시 빠진 항목이 없는지" 확인하는 용도로 활용했을 때 가장 유용하였다. 팀이 당연하게 생각하고 넘어간 부분을 제3자 시각에서 짚어주는 효과가 있었다.

**AI 활용에서 가장 어렵거나 주의가 필요했던 순간은?**
> AI가 제안하는 기능이나 항목이 프로젝트 규모·범위에 맞지 않는 경우가 많았다. 프롬프트에 프로젝트 맥락(팀 규모, 구현 기간, 과목 요구사항)을 충분히 설명하지 않으면 일반론적인 답변이 나온다는 것을 경험하였다.

**AI 없이 직접 해야 했던 작업은?**
> 팀원 역할 배분, 회의 내 의사결정, WBS 실제 주차 조율은 AI 없이 팀원들과 직접 논의해야 했다. AI는 팀의 역량, 일정, 팀원 간 의견 차이를 알 수 없기 때문이다.

**M2 단계에서는 AI를 어떻게 다르게 활용할 건가요?**
> 프롬프트를 작성할 때 과목 제출 가이드 내용이나 프로젝트 제약 조건을 함께 제공하면 더 적합한 답변을 얻을 수 있을 것 같다. M2 설계 단계에서는 Mermaid 문법 확인 등 기술적인 참고 용도로 활용해볼 계획이다.
