## 논의 내용

**안건 1. 설계 패턴 선정 논의**

논의 내용:
임태훈이 여러 디자인 패턴을 조사해서 우리 시스템에 맞는 것을 추렸다. 퀴즈 생성 시 과목별로 서로 다른 문제 목록을 만들어야 한다는 점에서 팩토리 계열 패턴이 적합하다는 의견을 냈다. 완전한 Factory Method 패턴은 클래스 수가 많아져서 프로젝트 규모에 비해 과도하다는 의견이 나왔고, `Quiz.generate_quiz()` 내부에 분기 처리를 두는 Simple Factory 방식으로 의견이 모였다. 박성주도 구현 측면에서 그게 훨씬 현실적이라고 동의했다.

결정 사항:
- 적용 패턴: Simple Factory (간이 팩토리)
- 적용 위치: `Quiz.generate_quiz()` 메서드 내부에서 과목별 분기 처리
- `main()` 함수에서는 과목 분기 로직 없이 `generate_quiz()` 호출만 수행

---

**안건 2. SOLID 원칙 검토 범위**

논의 내용:
SOLID 원칙 5가지 중 어디까지 해당하는지 논의했다. LSP는 현재 상속 관계가 없어서 "해당 없음"으로 처리하자는 의견이 나왔다. 안광우가 그냥 "해당 없음" 한 마디만 쓰면 이유 설명이 없어 설득력이 약하다고 지적했는데, 임태훈이 이유를 보완해서 쓰겠다고 했다. DIP, ISP 항목은 현재 설계에서 의미 있는 적용 사례가 없어 검토 범위에서 제외하기로 했다.

결정 사항:
- SRP, OCP, LSP 3가지 항목만 검토
- OCP 위반 사항은 솔직하게 기술하되, 개선 방향 명시
- LSP는 "해당 없음" + 이유 설명 포함

---

**안건 3. 구현 단계 준비 사항 점검**

논의 내용:
박성주가 구현 시작 전에 확인할 것들을 정리해왔다. Python 버전, 파일 구조, 클래스 네이밍 규칙 등이었다. `src/main.py` 단일 파일로 구현하고 표준 라이브러리만 사용하기로 확정했다.

결정 사항:
- 파일: `src/main.py` 단일 파일
- Python 표준 라이브러리만 사용 (외부 의존성 없음)
- 클래스명: PascalCase, 메서드명: snake_case (Python 관례 준수)
