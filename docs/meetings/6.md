## 논의 내용

**안건 1. 유스케이스 다이어그램 초안 검토**

논의 내용:
안병규가 Mermaid로 작성한 유스케이스 다이어그램을 같이 봤다. GitHub에서 렌더링이 잘 됐다. 유스케이스 번호와 관계 표현이 지난 회의 결정 사항과 일치하는지 확인했다. 안광우가 UC7(오답 다시 풀기)이 사용자에게 직접 연결되어 있는지 확인하고 싶다고 해서 다이어그램을 다시 보니 연결이 빠져 있어서 추가하기로 했다.

결정 사항:
- UC7(오답 문제 다시 풀기) → 사용자 연결 추가
- UC8(퀴즈 풀이 기록 확인) → 사용자 연결 확인 완료
- Mermaid 렌더링 정상 확인

---

**안건 2. 클래스 다이어그램 초안 검토**

논의 내용:
임태훈이 작성한 클래스 다이어그램 초안을 노트북으로 공유하며 같이 검토했다. 박성주가 `Quiz` 클래스의 `generateQuiz()` 메서드가 어떻게 동작하는지 설명해줬다. 이 시점에서 `Quiz` 클래스에 `answers` 속성이 빠져 있다는 점은 눈에 잘 띄지 않아서 넘어갔다(이후 인스펙션에서 발견됨). 전반적으로 클래스 구조와 요구사항 기능이 1대1 대응이 잘 된다는 평가.

박인찬이 GitHub에서 Mermaid 렌더링 시 한글 노드 이름 문제를 Claude에게 사전에 확인해봤는데, 최신 버전에서는 한글도 잘 렌더링된다는 것을 직접 테스트로 확인하였다.

결정 사항:
- 클래스 7개 확정 (User, Subject, Quiz, Question, Answer, WrongAnswerNote, QuizHistory)
- 속성·메서드 네이밍은 camelCase 통일 (다이어그램 표기 기준)
- 임태훈이 오늘 논의한 내용 반영 후 GitHub에 최종 업로드

---

**안건 3. 설계 패턴 및 SOLID 검토 일정 확인**

논의 내용:
다음 단계로 설계 패턴 적용 내역과 SOLID 원칙 검토를 해야 하는데, 이건 다이어그램이 확정된 후에 진행하는 것이 자연스럽다는 데 의견이 모였다. 임태훈이 UML 최종본이 나오면 본인이 바로 설계 패턴 문서를 작성하겠다고 했다.

결정 사항:
- 설계 패턴 문서 작성은 임태훈, 11주차 목표
- SOLID 원칙 검토도 임태훈 주도로 작성
- 안광우는 코드 구현 후 보안·품질 검토 의견 추가
