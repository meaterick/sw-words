# 📝 Coding Standards & Conventions

이 문서는 프로젝트의 코드 품질을 유지하고, 협업 효율성을 높이기 위한 코딩 표준을 정의합니다. 모든 기여자는 코드를 작성할 때 이 규칙을 준수해야 합니다.

---

## 1. 기본 원칙 (General Principles)

* 가독성 우선: 코드는 컴퓨터가 읽기 위해서가 아니라, 사람이 읽기 위해 작성합니다. 깨끗하고 명확한 코드가 현란한 코드보다 좋습니다.
* 보이스카우트 규칙: 캠프장은 처음 왔을 때보다 깨끗하게 해놓고 떠나야 합니다. 변경하는 파일의 기존 스파게티 코드는 조금씩이라도 개선하세요.
* YAGNI (You Ain't Gonna Need It): 지금 필요 없는 기능이나 과도한 추상화는 구현하지 않습니다.

---

## 2. 네이밍 규칙 (Naming Conventions)

변수, 함수, 클래스의 이름은 그 자체로 명확한 의도를 드러내야 합니다.

### 2.1. 표기법 (Casing)

| 대상 | 표기법 | 예시 |
| --- | --- | --- |
| 클래스 / 인터페이스 / 타입 | PascalCase | `UserProfile`, `AuthService` |
| 변수 / 함수 / 메서드 | camelCase | `totalPrice`, `calculateAmount()` |
| 상수 (Constants) | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT`, `API_URL` |
| 파일명 | kebab-case 또는 PascalCase | `user-profile.component.ts`, `AuthService.js` |

### 2.2. 의미 있는 이름 사용

* 함수: 동사로 시작하여 행동을 명확히 합니다. (예: `getUserData()`, `isValidEmail()`)
* 불리언(Boolean): 긍정형태의 의문문 구조를 사용합니다. (예: `isActive`, `hasToken`, `isCompleted`)
* 축약어 자제: 누구나 알 수 있는 축약어(`id`, `cnt` 등)가 아니라면 가급적 풀네임을 사용합니다. (예: `userRepository` > `userRepo`)

## 3. 코드 스타일 & 포맷팅 (Code Style & Formatting)

일관된 포맷팅을 위해 자동화 도구(Linter/Formatter) 사용을 강제합니다.

* 들여쓰기 (Indentation): 탭(Tab) 대신 2 Space를 사용합니다.
* 문장 끝 (Semicolon): 모든 문장의 끝에는 세미콜론(`;`)을 필수/제외 합니다. *(팀 내에서 택1)*
* 줄 바꿈 (Line Length): 한 줄은 최대 120자를 넘지 않도록 합니다.
* 중괄호 (Braces): K&R 스타일(Oman 스타일)을 따르며, 한 줄짜리 이프문이라도 중괄호 생략을 금지합니다.

```javascript
// ⭕ 올바른 예
if (isLoggedIn) {
  redirectToDashboard();
} else {
  showLoginError();
}

// ❌ 잘못된 예
if (isLoggedIn) redirectToDashboard();

```

## 4. 함수 및 메서드 설계 (Function Design)

* 단일 책임 원칙 (SRP): 하나의 함수는 단 하나의 일만 잘 수행해야 합니다.
* 함수 길이 최소화: 하나의 함수는 가급적 20줄을 넘지 않도록 작게 쪼갭니다.
* 매개변수(Arguments) 제한: 함수의 인자는 최대 3개 이하로 유지하세요. 그 이상이 필요하다면 객체(Object)나 DTO로 묶어서 전달합니다.

```typescript
// ❌ 인자가 너무 많음
function createUser(name, age, email, address, phoneNumber) { ... }

// ⭕ 객체로 묶어서 전달
interface UserDTO {
  name: string;
  age: number;
  email: string;
  address: string;
}
function createUser(userDoc: UserDTO) { ... }

```

## 5. 에러 핸들링 & 주석 (Error Handling & Comments)

### 5.1. 에러 처리

* 예외가 발생할 수 있는 곳은 반드시 `try-catch` 블록으로 감싸거나 적절한 에러 핸들러를 거쳐야 합니다.
* catch 블록을 비워두는 것(`catch(e) {}`)은 절대 금지합니다. 최소한 로깅이라도 남기세요.

### 5.2. 주석 (Comments)

* '무엇(What)'이나 '어떻게(How)'를 설명하는 주석은 피하세요. (그것은 코드가 말해줘야 합니다.)
* 주석은 왜 이렇게 작성했는지 배경이나 비즈니스 로직의 특수성을 설명할 때만 사용합니다.
* 할 일이 남았을 때는 `// TODO: 할 일 내용` 형식을 사용해 명시합니다.

## 6. Git & Pull Request 규칙

코딩 표준은 코드 작성뿐만 아니라 커밋과 리뷰 과정에도 적용됩니다.

* 커밋 메시지: [Conventional Commits](https://www.conventionalcommits.org/) 표준을 따릅니다.
* `feat:` 새로운 기능 추가
* `fix:` 버그 수정
* `docs:` 문서 수정
* `refactor:` 코드 리팩토링 (기능 변화 없음)


* PR 생성 규칙:
* 리뷰어가 이해할 수 있도록 PR 템플릿의 내용을 성실히 작성합니다.
* 최소 1명 이상의 Peer Reviewer의 승인(Approve)을 받아야 `main` 브랜치에 머지할 수 있습니다.
